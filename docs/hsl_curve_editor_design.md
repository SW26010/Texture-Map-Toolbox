# HLS 曲线编辑器 — 设计与性能分析

## 功能概述

交互式工具，对亮度-颜色 LUT 进行 HLS 曲线调整，实时预览重着色效果。

**当前实现**：[`scripts/hsl_curve_editor.py`](../scripts/hsl_curve_editor.py)（matplotlib 版）

---

## 性能分析（4096×4096 图，120Hz 显示器）

### 优化历程

| 版本 | recolor | draw | 总计 | 说明 |
|------|---------|------|------|------|
| 初始版（float LUT + 全尺寸）| 788 ms | 52 ms | **840 ms** | 原始实现 |
| + uint8 LUT + 25% 缩图 | 6.5 ms | 3.4 ms | **10.2 ms** | ×82 加速 |
| + `np.take` + 预分配 buffer | 4.7 ms | 3.4 ms | **8.4 ms** | ×1.2 加速 |

### 当前瓶颈（8.4ms/帧）

| 阶段 | 耗时 | 瓶颈原因 |
|------|------|----------|
| `apply_hsl_curves` + LUT 量化 | 0.3 ms | 仅 256 点，已到极限 |
| `recolor`（`np.take` + 掩码写零）| 4.7 ms | 随机 gather 访问，内存带宽极限 |
| `draw_idle`（matplotlib 渲染）| 3.4 ms | 1024×1024 图像渲染 + 调度延迟 |

### 理论分析

```
gather 理论最小耗时：7 MB 内存流量 / 20 GB/s ≈ 0.35 ms
当前实测：4.7 ms ≈ 理论值的 13×
差距来源：随机地址访问模式，CPU 无法向量化预取
```

**结论**：纯 numpy 已到极限，进一步优化需跳出 numpy。

### 感知帧率问题

- 理论帧率：1000 / 8.4 ≈ **119 fps**
- 实际感知：**≈60-80 fps**（仍有轻微卡顿）
- 原因：8.4ms 总耗时 > 120Hz 帧预算（8.33ms），`draw_idle` 的异步调度进一步增加延迟抖动

## 进一步优化方向（未实施）

| 方案 | 预期收益 | 适用条件 |
|------|----------|----------|
| **numba `@njit(parallel=True)`** | recolor 4.7ms → ~1ms，总计 ~5ms，稳定达到 120fps | 需要解决 120fps 卡顿 |
| **matplotlib blitting** | draw 3.4ms → ~0.5ms | 配合 numba 使用，收益更明显 |
| **GPU（CuPy）** | gather 极快，但 CPU↔GPU 传输 ~0.5-1ms 抵消收益 | 不推荐，性价比低 |

### 候选框架对比

| 框架 | 先进性 | 美观度 | 性能 | 实现难度 | 备注 |
|------|--------|--------|------|----------|------|
| **Dear PyGui** | ★★★★★ | ★★★★ | ★★★★★ | ★★★ | GPU 渲染，内置拖拽控制点，可解决 120fps 问题 |
| **PyQt6** | ★★★ | ★★★★ | ★★★★ | ★★★ | 可嵌入 matplotlib，原生文件对话框，生态最完整 |
| **NiceGUI** | ★★★★★ | ★★★★★ | ★★★ | ★★★ | Web UI，实时图像传输延迟不适合高频拖拽 |
| **Flet** | ★★★★ | ★★★★★ | ★★★ | ★★★★ | Flutter for Python，图像控件生态弱 |
| **wxPython** | ★★ | ★★★ | ★★★★ | ★★★ | Win32 真原生，API 偏旧 |

### Windows 原生说明

Python 没有官方的 WinUI 3 绑定。"最原生"的选项：
- **wxPython**：直接调用 Win32 控件，外观与系统完全一致
- **PyQt6 + pyqtdarktheme**：跟随系统主题，Fluent Design 风格，开发效率更高

### 当前结论

**待定**，优先完成算法讨论。候选方向：

1. **Dear PyGui** — 如果 120fps 流畅度是优先级，内置拖拽点省开发量
2. **PyQt6** — 如果需要与系统融合 + 嵌入 matplotlib 分析图表
